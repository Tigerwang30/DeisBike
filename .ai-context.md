## Critical Env Vars

| Name | Purpose | Expires? |
|------|---------|---------|
| `NODE_ENV` | `development` enables relaxed local settings | No |
| `LINKA_API_BASE_URL` | `https://app.linkalock.com/api/merchant_api` | No |
| `LINKA_ACCESS_TOKEN` | Meteor session token (X-Auth-Token header) | Yes — daily |
| `LINKA_USER_ID` | Meteor user ID (X-User-Id header) | Yes — daily |
| `STORAGE_BACKEND` | `json` (default) or `memory` | No |

LINKA_ACCESS_TOKEN and LINKA_USER_ID are NOT in bikes.json — they are fleet-level credentials in `.env`. Per-bike credentials (linka_token, mac_addr) live in `bikes.json`.

## Data Models

### Session (active ride, in store["sessions"])
```python
{
  "sessionId": "session-{timestamp}-{bikeId}",   # str, primary key
  "bikeId":    "5000",                           # str
  "startTime": "2024-01-15T10:30:00",            # ISO UTC string
  "status":    "ride_active",                    # "ride_active" | "chain_unlocked"
  # Two-step flow only:
  "chainUnlocked": True,
  "wheelUnlocked": False,
}
```

Rides are not persisted to history once locked — `_finalize_ride` locks the bike and deletes the session; there is no per-rider record.

## API Endpoint Inventory

```
GET  /api/bikes                   List all bikes from registry
GET  /api/bikes/locations/all     List bikes with lat/lng/name/location
GET  /api/bikes/{bike_id}         Single bike status (chainLocked, battery)

POST /api/command                 actions: open | lock | unlock_chain | unlock_wheel | status
POST /api/command/webhook         LINKA webhook; event: chain_locked

GET  /api/rides/active            The single currently-active session (or { active: false })
```

No auth on any endpoint — the app is fully open.

## LINKA Integration Facts

- Base URL: `https://app.linkalock.com/api/merchant_api`
- All requests: POST (with body) or GET (status check)
- Required headers: `X-Auth-Token`, `X-User-Id`, `Origin: https://fleetview.linkalock.com`, `Referer: https://fleetview.linkalock.com/`
- Command body (per bike, from `registry.command_body(bike_id)`):
  ```json
  { "access_token": "<linka_token>", "mac_addr": "<mac>", "schedule": true, "firmware_version": "2.6.15", "smartkey_mac": "" }
  ```
- Endpoints used: `/command_unlock`, `/command_lock`, `/device_status/{bike_id}`
- Error codes: 200=queued (OK), 620=another command in progress (wait 5-10s), 504=expired credentials
- Execution: commands queue in LINKA cloud; execute when LINKA app on phone comes within Bluetooth range of lock
- `call_linka()` returns simulated success if `LINKA_API_KEY` env var is not set (dev mode)

## Store Interface Contract

```python
# All implemented by MemoryStore and JsonFileStore
get_session(session_id: str) -> dict | None
get_active_session() -> tuple[str, dict] | None    # first session with status == "ride_active"
get_all_active_sessions() -> dict[str, dict]
put_session(session_id: str, data: dict) -> None
delete_session(session_id: str) -> None
```

Access via `get_store()` from `api/storage/__init__.py` — never import `MemoryStore`/`JsonFileStore` directly.

## Business Rules

1. No auth or approval gate — any visitor can unlock any available bike
2. Only one active ride globally at a time (enforced by `get_active_session` returning the first match)
3. Bikes are identified by string ID (e.g. `"5000"`) — always string, never int
4. Session IDs format: `"session-{unix_ms}-{bike_id}"`
5. Ride duration is computed server-side in minutes (integer); client displays seconds locally
6. Completed rides are not persisted anywhere — locking a bike deletes its session with no history record

## Non-obvious Gotchas

1. **Phone required.** LINKA lock has no WiFi. Commands queue in cloud, execute only when LINKA app on phone is within Bluetooth range. If no phone is near, the lock command succeeds (API 200) but the lock does not move.

2. **LINKA credentials expire daily.** `LINKA_ACCESS_TOKEN` and `LINKA_USER_ID` are Meteor.js session tokens. They expire in ~24 hours. Rotate via FleetView DevTools → Network tab → copy x-auth-token and x-user-id from any POST request.

3. **Vercel kills in-process state.** Each Vercel function invocation may run in a fresh process. `JsonFileStore` writes to `/tmp` which is ephemeral on Vercel. Set `STORAGE_BACKEND=memory` on Vercel or use external persistence.

4. **`call_linka` simulates if no API key.** If `LINKA_API_KEY` is not set, `call_linka()` returns `{"success": True, "simulated": True}` without making any HTTP request. This enables frontend development without real credentials.

5. **`bikes.json` is gitignored.** Contains per-bike LINKA tokens. Commit `bikes.example.json` instead. New developers copy it to `bikes.json` and fill in real values.

6. **Two-step unlock flow is lightly used.** `unlock_chain` + `unlock_wheel` are in `command.py` as thin inline logic (not extracted to `RideService`). The `open` action (single-step) is preferred and used by the current UI. The two-step flow exists for future hardware that distinguishes chain from wheel locking.

7. **`MapPage` uses `bikeService.getAll()`** (returns name, location, available) not `getLocations()` (returns lat/lng only). Both endpoints exist; `getAll` is the right one for the card grid.

8. **`RideContext` always polls.** With no auth/approval gate, `useRideStatus(true)` runs unconditionally — there's no user state to gate it on.

9. **`data/store.json` parent dir must exist.** `JsonFileStore._save()` calls `os.makedirs(os.path.dirname(path), exist_ok=True)`. The `data/` directory is gitignored and created automatically on first write.

## Import Conventions

- Backend routes import services via module singletons: `from api.services.ride_service import ride_service`
- Routes never import `_state.py` (deleted) — always use `get_store()`
- Routes never import `_command_body()` from `_linka.py` — always use `registry.command_body(bike_id)`
- Frontend pages import from specific service files: `import { commandService } from '../services/commands'`
- Frontend pages never import from `'../services/api'` — that monolith is superseded by per-domain files
